📖 ISTQB Learning Resources

Quick reference guides, glossary, formulas, and exam preparation tips

📚 Comprehensive Glossary (50+ Terms)

Acceptance Testing
Testing to determine if the system meets user needs and acceptance criteria. Performed by users or their representatives.
Test Case
A set of input values, execution preconditions, expected results, and execution postconditions to test specific functionality.
Defect
A flaw or fault in a work product that may cause the software to fail or deviate from specified requirements.
Failure
The inability of software to perform a required function or deviation from specified behavior.
Error
A human mistake or action that introduces a defect into the software.
Root Cause
The fundamental reason why an error was made; identified through root cause analysis.
Test Coverage
A measure of the extent to which the test object has been exercised by tests, expressed as a percentage.
Regression Testing
Testing of previously tested functionality to ensure that changes have not introduced new defects.
Smoke Testing
A non-exhaustive test to ensure that critical functionality works, determining if a build is stable enough for testing.
Sanity Testing
Testing a specific function or defect fix in isolation to verify it works as intended.
Equivalence Partitioning
A test technique that divides input data into partitions where all values behave similarly.
Boundary Value Analysis
A test technique focusing on testing at the boundaries of input domains where defects often occur.
Decision Table
A table showing combinations of conditions and corresponding actions for testing business logic.
State Transition Diagram
A diagram showing states of a system and transitions between them, used for test design.
Use Case
A description of interactions between actors and a system to achieve specific goals, used for test design.
Test Condition
An aspect of a work product that can be verified by one or more test cases.
Verification
"Are we building the product right?" - Checking conformance to specifications and requirements.
Validation
"Are we building the right product?" - Checking conformance to user needs and intended use.
Static Testing
Testing without executing the software, including reviews, inspections, and static analysis.
Dynamic Testing
Testing that involves executing the software to observe behavior and identify failures.
Review
A technique where one or more people examine a work product to find defects and provide feedback.
Inspection
A formal, documented peer review process with defined roles, process, and metrics.
Walkthrough
An informal static technique where the author leads a structured review of the work product.
Technical Review
A formal evaluation by peers to assess correctness and compliance with standards.
Static Analysis
Automated analysis of code without execution to identify defects and quality issues.
White-Box Testing
Testing using internal code knowledge to design tests, achieving high code coverage.
Black-Box Testing
Testing without code knowledge, based on inputs and outputs, tests the specification.
Test Level
A specific stage of testing in the SDLC (component, integration, system, acceptance).
Test Type
The focus of testing (functional, non-functional, regression, confirmation, etc.).
Component Testing
Testing individual software components in isolation, typically by developers (unit testing).
Integration Testing
Testing interactions between integrated components to verify correct communication.
System Testing
Testing the complete integrated system against functional and non-functional requirements.
Risk-Based Testing
Prioritizing testing based on identified risks, focusing effort on high-risk areas.
Test Automation
Using tools to execute tests automatically, improving efficiency and consistency.
CI/CD
Continuous Integration and Continuous Deployment; automated build, test, and deployment pipeline.
Test Environment
A system environment in which tests are executed, separate from production.
Test Data
Input data used for executing test cases, typically created to cover various scenarios.
Test Metric
A quantitative measure used to monitor and track testing progress and effectiveness.
Defect Tracking
The process of recording, monitoring, and managing defects from discovery to resolution.
Entry Criteria
Conditions that must be met before testing can begin (e.g., test environment ready).
Exit Criteria
Conditions that must be met for testing to be considered complete (e.g., planned tests executed).
Test Plan
A detailed document defining test activities, resources, schedule, and deliverables.
Test Strategy
High-level approach defining overall testing philosophy and scope.
Performance Testing
Testing to measure system responsiveness, stability, and scalability under various loads.
Security Testing
Testing to identify vulnerabilities and verify protection against unauthorized access.
Usability Testing
Testing to evaluate ease of use and user experience of the software.

📐 Formulas & Checklists

⏱️ PERT (Program Evaluation and Review Technique)
Expected Time = (Optimistic + 4 × Most Likely + Pessimistic) / 6
Used for project scheduling and risk estimation. Weights the most likely estimate more heavily. Useful for test estimation with uncertainty.
🎯 Risk Assessment Formula
Risk Level = Probability × Impact
Combines likelihood of risk occurring with severity of impact. Helps prioritize testing efforts on high-risk areas.
📊 Test Coverage Formula
Coverage (%) = (Items Tested / Total Items) × 100
Measures percentage of code, requirements, or paths covered by testing. Common targets: 80% code coverage, 90% requirement coverage.
🐛 Defect Density Formula
Defect Density = Number of Defects / Size of Work Product
Indicates quality of a work product. Helps identify problem areas and predict remaining defects.

✅ Test Planning Checklist

  • Define test objectives and scope
  • Identify test levels and types needed
  • Determine test approach and strategy
  • Estimate resources and effort
  • Establish test schedule and milestones
  • Define entry and exit criteria
  • Identify risks and mitigation plans
  • Plan test environment setup
  • Define test data requirements
  • Plan resource allocation
  • Establish communication plan
  • Define test deliverables

✅ Test Case Design Checklist

  • Identify test conditions from requirements
  • Apply appropriate test techniques
  • Cover positive scenarios (happy path)
  • Cover negative scenarios (error cases)
  • Test boundary values
  • Test data combinations
  • Include preconditions and setup
  • Define clear expected results
  • Include postconditions and cleanup
  • Review test cases for clarity
  • Ensure traceability to requirements
  • Avoid duplicate test cases

💡 Exam Preparation & Tips

📋
ISTQB CTFL Exam Details
Duration: 60 minutes | Questions: 40 multiple choice | Pass Score: 65% (26/40) | Languages: Multiple
🎯 Exam Strategy
Time Management: 90 seconds per question (~1.5 min). Don't spend too long on difficult questions; mark and return.
Reading Carefully: Read questions completely; look for "NOT", "EXCEPT", "BEST".
Question Distribution: Ch1 (10%), Ch2 (15%), Ch3 (10%), Ch4 (20%), Ch5 (20%), Ch6 (5%), SDLC (20%)
📚 Study Strategy
Start Early: Begin 4-6 weeks before exam.
Learn Concepts: Understand testing principles, not just memorize.
Practice Questions: Take unlimited practice quizzes to identify weak areas.
Review Weak Areas: Focus on K2 and K3 concepts that require application.
Terminology: Master ISTQB glossary; exam uses precise terminology.
🧠 Tricky Concepts to Master
Test vs QA: Testing finds defects; QA prevents defects through processes.
Verification vs Validation: Verification = against specs; Validation = against user needs.
Error vs Defect vs Failure: Error (human mistake) → Defect (flaw in code) → Failure (incorrect behavior).
Static vs Dynamic: Static = no execution (reviews); Dynamic = with execution.
Test Levels: Component (developers), Integration (mixed), System (testers), Acceptance (users).
✍️ Sample Exam Question & Strategy
Question: "Which of the following is a main objective of testing?"
Options: A) Fix all defects, B) Reduce risk, C) Guarantee quality, D) Prove software works
Strategy: Look for realistic answer. A is wrong (testing doesn't fix). C is too absolute (can't guarantee). D is wrong (testing shows presence, not absence). B is correct (risk reduction is a goal).
Key Principle: Testing shows defects are present; cannot prove absence of defects.

💪 Success Tips

  • Understand "why" not just "what"
  • Learn by doing (practice tests)
  • Master K2 and K3 concepts (understand + apply)
  • Focus on test techniques and their applications
  • Understand testing within SDLC context
  • Study with other learners for discussion
  • Take full-length practice exams
  • Review explanations for incorrect answers
  • Manage exam anxiety through preparation
  • Arrive early on exam day, relax and focus
