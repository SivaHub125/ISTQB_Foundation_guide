📚 Fundamentals of Testing

🎯 What is Testing? K1

Software testing assesses software quality and helps reduce the risk of software failure. Testing is a set of activities to discover defects and evaluate quality of software work products.

🔍 Verification

"Are we building the product right?"
Checking against specified requirements

✅ Validation

"Are we building the right product?"
Checking against user needs

🔄
Dynamic Testing: Execution of software
📋
Static Testing: No execution (reviews + static analysis)

→

📐 Seven Testing Principles K2

1️⃣

Presence, Not Absence

Testing shows defects are present, not that there are none

2️⃣

Exhaustive Testing Impossible

Risk-based approach is practical

3️⃣

Early Testing Saves Money

Find defects early in SDLC

4️⃣

Defect Clustering

Most defects in few modules (Pareto principle)

5️⃣

Tests Wear Out

Repeated tests become less effective (Pesticide Paradox)

6️⃣

Context Dependent

No single universal approach

7️⃣

Absence-of-Defects Fallacy

Bug-free ≠ success; must meet user needs

⚫ Black-Box Test Techniques

1. Equivalence Partitioning (EP) K3

Divide input data into groups that behave similarly. Test ONE representative from each partition.

Example: Age Input (18-65 valid)

Partitions:
• Invalid: < 18 (one test)
• Valid: 18-65 (one test)
• Invalid: > 65 (one test)

2. Boundary Value Analysis (BVA) K3

Test values at boundaries. Most defects found at boundaries!

Example: Age 18-65

Test: 17, 18, 65, 66 (boundary values)

3. Decision Table Testing K3

Systematically test combinations of conditions and actions.

Format

Columns: Conditions (T/F) × Actions → Test cases

4. State Transition Testing K3

Test system behavior as it moves between states.

Example: Login System

Logged Out → Logged In → Logged Out

🔵 White-Box Test Techniques

1. Statement Testing & Coverage K2

Execute every statement at least once.

Coverage Formula

Statement Coverage = (Executed Statements / Total Statements) × 100

2. Branch Testing & Coverage K2

Execute each branch direction (true and false) at least once.

Coverage Formula

Branch Coverage = (Executed Branches / Total Branches) × 100

3. Value of White-Box Testing

✅
Identifies hidden defects
📐
Measures structural quality
🔍
Finds dead code

Product Risk Response Strategies

🔄

Risk Transfer

Shift risk to another party (e.g., buying insurance, outsourcing, warranties, guarantees)

Example: Use third-party testing service, contractual risk assignment

✅

Risk Acceptance

Accept the risk when it's within acceptable limits. Decide it's not worth the cost to mitigate.

Example: Low-risk areas with minimal impact may be accepted

🛡️

Risk Mitigation

Take actions to reduce probability and/or impact of risk. Most common response strategy.

Example: More testing on high-risk features, code reviews, early testing

📋

Contingency Plan

Prepare actions to be executed if the risk occurs. A backup plan for high-impact risks.

Example: Extra testing budget, alternative suppliers, fallback release date

💡 Risk Response Priority

Mitigate (most common) → Transfer → Avoid → Accept (last resort)
